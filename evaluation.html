    Evaluation of Personas

    Personas are descriptions of fictitious people who have a set of traits and requirements that you want to design for. They are ideal for understanding the user requirements, and getting an idea of the target audiences’ characteristics.

    The advantages of using a Persona is that it is an efficient way to express design requirements. More importantly, it helps you to see users as people rather than a set of design requirements. Through this, we get an in-depth idea of how a potential user(s) would be.

    However, there are cons too. Personas are considered to be less scientific. The number of personas which can be used are limited too so designing them well is crucial. The major drawback is that it can be difficult for some developers to make use of them. This is because design requirements are more functional whereas, personas are more conceptual. Nevertheless, it is still beneficial to design them during the initial stage of requirement gathering.

    Evaluation of Scenario

    Scenarios are helpful to describe the human activities which take place in a story format. It allows us to explore and observe the context, needs, and requirements.

    The advantage is that this procedure is extremely natural. This means that we can identify the users’ needs during their realistic day-to-day lives. More specifically, our scenario helped us to visualise a situation during which a user might want to book or cancel a class booking in a practical sense.

    However, the disadvantage of ‘Scenarios’ is that it is more focused on task characteristics. However, it fails to outline the details of the interface design and layout of the Amazon Echo model. So we would need another method which points out the design and layout.
